## Objective

Replace the FTS5 `unicode61` tokenizer with `trigram` for autocomplete-optimized substring matching. Current search requires users to know the start of a drug name (`dolipr*`). Trigram enables `dol` → `Doliprane` without wildcards — the single highest-impact UX improvement for the search API.

Purpose: Autocomplete on partial strings, not just prefixes
Output: FTS5 uses trigram tokenizer, short-query fallback via LIKE

## Context

@src/db/fts.rs
@src/api/ (search endpoint)

## Decision: rusqlite Bundled SQLite Supports Trigram

rusqlite ships SQLite3.48+ via the `bundled` feature (default). Trigram was added in SQLite 3.48.0. No compile flags needed — just change the tokenizer in SQL. The only caveat: deployment targets using system SQLite older than 3.48 need rusqlite's bundled build.

## Decision: Option A — Replace unicode61 with trigram

Two FTS5 tables (one per tokenizer) adds complexity for marginal benefit. With 15K rows and primarily autocomplete use cases, replacing unicode61 with trigram is sufficient. Keep unicode61 for phrase/prefix queries via the BM25-ranked API endpoint.

Files: src/db/fts.rs
Action: Change the `tokenize` parameter in `create_fts_tables`:

In the `CREATE VIRTUAL TABLE` statement, change:
```sql
tokenize='unicode61 remove_diacritics 1'
```
To:
```sql

```

This is a one-line change in `fts.rs`. Since `create_fts_tables` drops and recreates the table on every ingest, the new tokenizer takes effect immediately.

### Task 2: Add short-query fallback in search API
Files: src/api/ (search endpoint handler)
Action: FTS5 trigram requires minimum 3-character substrings. For queries < 3 chars, fall back to LIKE on the `name` column:

In the search endpoint, before the FTS5 MATCH query:
```rust
if query.chars().count() < 3 {
    // Short query: use LIKE for prefix match
    let results: Vec<Drug> = conn.query_rows(
        "SELECT d.* FROM drugs d WHERE d.name LIKE ?1 || '%' LIMIT 20",
        [query.to_uppercase()],
    )?;
    return Ok(results);
}
// Normal query: use FTS5 trigram MATCH
let fts_results: Vec<Drug> = conn.query_rows(
    "SELECT d.* FROM drugs_fts fts JOIN drugs d ON d.cis = fts.cis
     WHERE drugs_fts MATCH ?1 ORDER BY bm25(drugs_fts) LIMIT 20",
    [query],
)?;
```

Verify: `curl "http://localhost:8080/drugs/search?q=dol"` returns Doliprane results; `curl "http://localhost:8080/drugs/search?q=ib"` returns Ibuprofen results via LIKE fallback

### Task 3: Verify trigram availability at runtime
Files: src/db/fts.rs or src/main.rs
Action: Add a runtime check that trigram is available before creating the FTS5 table. If trigram is not available (system SQLite< 3.48), fall back to unicode61:

```rust
fn trigram_available(conn: &Connection) -> bool {
    let result: Result<String, _> = conn.query_row(
        "PRAGMA compile_options",
        [],
        |r| r.get(0),
    );
    // Check if SQLite version >= 3.48 (3048000)
    let version_num = rusqlite::version_number();
    version_num >= 3_048_000
}
```

In `create_fts_tables`, use the appropriate tokenizer:
```rust
let tokenizer = if trigram_available(conn) {
    "trigram remove_diacritics 1"
} else {
    "unicode61 remove_diacritics 1"
};
let sql = format!(
    "CREATE VIRTUAL TABLE IF NOT EXISTS drugs_fts USING fts5(
        cis, name_raw, name, atc_code, form, lab_name, substance_name,
        tokenize='{}'
    )",
    tokenizer
);
```

Verify: `cargo test --lib` passes; fallback to unicode61 works when trigram unavailable
Done: Runtime tokenizer selection based on SQLite version.
