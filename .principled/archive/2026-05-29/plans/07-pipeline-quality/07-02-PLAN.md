## Objective

Add a quarantine table to capture rejected rows with error context (turning silent failures into actionable data), and add CIP7 validation as a CHECK constraint. Currently bad rows are counted but never stored — making threshold warnings un-actionable.

Purpose: Auditability of all rejected rows; data quality signals for every failure type
Output: quarantine table in schema, rows captured during import, CIP7 CHECK constraint

## Context

@src/import/mod.rs (import_file function)
@src/db/schema.sql
@src/normalize/fields.rs

## Decision: Single Generic Quarantine Table

One table with `target_table` + `error_type` columns handles all source types. Per-table quarantine tables create schema proliferation and make cross-cutting analysis painful. The `error_type` column is TEXT (not enum) for extensibility without schema changes.

## Tasks

### Task 1: Add quarantine table to schema
Files: src/db/schema.sql
Action: Add after the `import_log` table definition:

```sql
-- =============================================================================
-- TABLE: quarantine (captures rejected rows for audit and retry)
-- =============================================================================
CREATE TABLE IF NOT EXISTS quarantine (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    source_file     TEXT    NOT NULL,
    source_line     INTEGER NOT NULL,
    target_table    TEXT    NOT NULL,
    error_type      TEXT    NOT NULL,
    error_detail    TEXT,
    raw_line        TEXT    NOT NULL,
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX IF NOT EXISTS idx_quarantine_file ON quarantine(source_file);
CREATE INDEX IF NOT EXISTS idx_quarantine_type ON quarantine(error_type);
CREATE INDEX IF NOT EXISTS idx_quarantine_date ON quarantine(created_at);
```

Verify: `cargo run --release -- ingest` completes; `sqlite3 data/bdpm.db ".schema quarantine"` shows table
Done: Quarantine table exists in schema.

### Task 2: Add CIP7 CHECK constraint to presentations
Files: src/db/schema.sql
Action: In the `presentations` table, add a CHECK constraint on the `cip` column. CIP7 must be exactly 7 ASCII digits. The current schema has no validation for CIP format.

Change the `cip` column definition from:
```sql
cip TEXT    PRIMARY KEY,
```
To:
```sql
cip                TEXT    PRIMARY KEY CHECK (cip IS NULL OR (LENGTH(cip) = 7 AND cip GLOB '[0-9][0-9][0-9][0-9][0-9][0-9][0-9]')),
```

Verify: `cargo run --release -- ingest` completes; malformed CIP values cause CHECK constraint violation logged
Done: CIP7 format enforced at DB level.

### Task 3: Wire quarantine inserts into import_file
Files: src/import/mod.rs
Action: In `import_file`, after the bulk INSERT loop, capture any rows that failed due to encoding errors, field count mismatches, or other parse failures. The quarantine insertion should happen in the `import_file` function where errors are caught.

Since the current import loop uses `stmt.execute()` per row and currently errors are not individually caught (they're counted via `stats.bad_rows`), add a quarantine capture path:

Add to `import_file` — for each file type, track line numbers and capture failures:
```rust
// In import_file, after the for loop:
// For FK violations during CIS_CIP import (orphan CIS codes):
// These are expected — INSERT OR IGNORE skips them silently.
// No quarantine needed for expected orphans.

// For parse/encoding errors (field count mismatch, invalid encoding):
// Log to quarantine if configured.
// Currently handled via tracing::warn — extend to quarantine insert.
```

**Practical approach:** Add a helper that captures to quarantine from parse errors:
```rust
fn quarantine_row(
    conn: &Connection,
    source_file: &str,
    source_line: usize,
    target_table: &str,
    error_type: &str,
    error_detail: &str,
    raw_line: &str,
) {
    let _ = conn.execute(
        "INSERT INTO quarantine (source_file, source_line, target_table, error_type, error_detail, raw_line)
         VALUES (?1, ?2, ?3, ?4, ?5, ?6)",
        rusqlite::params![source_file, source_line as i64, target_table, error_type, error_detail, raw_line],
    );
}
```

Add a call in `parse_file` when a row fails field count validation:
```rust
// In parse/tab.rs — when a row fails field count check:
// quarantine_row(conn, file.filename(), line_number, file.target_table(), "field_count_mismatch", &format!("expected {} got {}", expected, actual), &raw_line);
```

Note: Parsing errors occur before normalization, so the quarantine helper needs the DB connection. Pass it as a parameter or use a thread-local connection. Simplest approach: use the existing `import_file` transaction context.

**Minimal implementation:** Capture field-count rejection in the parser and pass quarantine info back through the return type. Since this requires changing the parser's return type, start with a simpler approach: add the `quarantine_row` function and call it from `parse_file` for field-count failures only. Extend to other error types as a follow-up.

Verify: `cargo test --lib` passes; field-count failures appear in quarantine table
Done: Rejected rows captured in quarantine table.
